# Merge Node — нода объединения данных

## Концепция

Нода с **двумя входными портами** (как в n8n). Принимает данные из двух веток, объединяет по выбранному режиму, отдаёт результат дальше.

## Визуал на канвасе

```
[Ветка A] ──→ ●
               │ MERGE │──→ [Следующая нода]
[Ветка B] ──→ ●
```

Два target handle (вход сверху и снизу), один source handle (выход).

## Режимы (modes)

| Режим | Описание | Аналог |
|-------|----------|--------|
| **Append** | Склеить два массива в один | `list_a + list_b` |
| **Combine by field** | JOIN по общему полю | SQL INNER JOIN |
| **Keep matches** | Только совпадения (пересечение) | SQL INNER JOIN без доп. полей |
| **Keep non-matches** | Элементы из A которых нет в B | SQL LEFT ANTI JOIN |
| **Combine by position** | Объединить по индексу (zip) | Python `zip(a, b)` |

## Панель свойств

```
┌─────────────────────────────────────────┐
│ Режим: [▼ Append]                       │
│                                         │
│ Вход A: (автоматически из связи)        │
│ Вход B: (автоматически из связи)        │
│                                         │
│ ── Для "Combine by field": ──           │
│ Поле A: [user_id        ]              │
│ Поле B: [telegram_id    ]              │
│                                         │
│ Сохранить в: [merged_results]           │
└─────────────────────────────────────────┘
```

## Примеры использования

### 1. Сравнение курсов (боты + сайты)
```
[Опрос 16 ботов] ──┐
                    ├──→ [Merge: Append] → [json_format с сортировкой]
[HTTP запросы к 9 сайтам] ─┘
```
Результат: единый рейтинг из 25 обменников, отсортированный по выгодности.

### 2. Рассылка по сегментам
```
[Запрос VIP из БД] ──────┐
                          ├──→ [Merge: Append] → [Дедупликация] → [Broadcast]
[Запрос активных за 7д] ─┘
```

### 3. Модерация — чёрный список
```
[Новые участники группы] ──┐
                            ├──→ [Merge: Keep matches] → [Бан]
[Чёрный список из БД] ─────┘
```
Банит только тех новичков, кто есть в чёрном списке.

### 4. Синхронизация профилей
```
[Пользователи Telegram] ──┐
                           ├──→ [Merge: Combine by field (user_id)] → [Обновить]
[Данные из Google Sheets] ─┘
```
Объединяет данные из двух источников по общему полю.

### 5. Параллельный парсинг
```
[Парсинг сайта A] ──┐
                     ├──→ [Merge: Append] → [Фильтрация] → [Сохранить]
[Парсинг сайта B] ──┘
```

### 6. Мультибот статистика
```
[Бот 1: заказы за день] ──┐
                           ├──→ [Merge: Append] → [Подсчёт суммы]
[Бот 2: заказы за день] ──┘
```

## Архитектурные требования

### Канвас (React Flow)
- Поддержка нод с 2+ target handles
- Визуальное отличие merge-ноды (другая форма или цвет)
- Валидация: оба входа должны быть подключены

### Граф переходов
- Нода "ждёт" пока обе входящие ветки завершатся
- Нужен механизм синхронизации (barrier/join point)
- Варианты: 
  - Последовательное выполнение (A потом B потом merge)
  - Параллельное (asyncio.gather для A и B, потом merge)

### Генератор кода
```python
# Вариант: последовательное выполнение
_merge_input_a = user_data[user_id].get("branch_a_results", [])
_merge_input_b = user_data[user_id].get("branch_b_results", [])

# Append
merged = _merge_input_a + _merge_input_b

# Combine by field
merged = []
for a in _merge_input_a:
    for b in _merge_input_b:
        if a.get("user_id") == b.get("telegram_id"):
            merged.append({**a, **b})
```

### Сохранение состояния
- Флаг "ветка A завершена" / "ветка B завершена"
- Merge выполняется только когда оба флага True
- Сброс флагов после выполнения

| Компонент | Время |

| Канвас: 2 target handles | 1 день |

| Синхронизация веток | 1-2 дня |
| Тесты + документация | 1 день |
| **Итого** | **5-6 дней** |

## Промежуточное решение (MVP)

Пока merge-нода не реализована, используется mode `array_concat` в set_variable:
```json
{
  "mode": "array_concat",
  "value": "bot_compare_results",
  "variable": "all_results",
  "concatWith": "compare_results"
}
```
Это покрывает 90% кейсов (Append) без изменения архитектуры канваса.

## Приоритет

Средний. Реализовать после:
- Параллельного выполнения (asyncio.gather для userbot-цепочек)
- Кэширования результатов

Реализовать до:
- Расширенных интеграций (webhooks, внешние API)
