# Вкладка «Агент» — управление токенами MCP (per-user PAT)

## Дата: 2026-06-25

Самообслуживаемая вкладка, где любой пользователь генерирует персональные токены агента (Personal Access Token) и подключает ИИ-агента (Kiro / Cursor / Claude Desktop) к своим проектам через MCP — без ручного SQL. Модель GitHub PAT / n8n API-ключ.

> **Это UI-часть Фазы 0.2 (этап 0.2b)** плана `../mcp/mcp-live-editing.md`. Серверная инфраструктура токенов (`agent_tokens`, resolver `identifyAgent`, deny-by-default) уже сделана в 0.2a (коммит `8dd17ba23`). Целевая модель безопасности — `../infrastructure/api-security-ideal-architecture.md` (слой 4, источник личности PAT).

---

## Зачем

Сейчас токен агента можно завести только напрямую в БД (через SQL). Вкладка «Агент» превращает это в self-service: пользователь сам создаёт/отзывает токены и получает готовый конфиг MCP. Это и есть точка входа «каждый человек может подключить MCP».

---

## Где живёт

Отдельная вкладка «Агент» (или «ИИ-агент / MCP») в настройках аккаунта — рядом с профилем и токенами ботов. Раздел личный: токены привязаны к `ownerId` текущего пользователя.

---

## Состав вкладки

### Блок 1 — Онбординг (2–3 строки)
Короткое пояснение: «Подключи ИИ-агента к своим проектам. Агент сможет создавать и редактировать ботов в реальном времени прямо на холсте». Ссылка «Как настроить» → инструкция.

### Блок 2 — Список токенов
Таблица существующих токенов (из `agent_tokens`, **без секрета**):

| Поле | Источник | Назначение |
|------|----------|-----------|
| Название | `label` | различать токены («Cursor на ноуте», «Kiro рабочий») |
| Префикс | `prefix` | `mcp_qQqx9Aeu…` — узнать токен, не используя его |
| Права | `scopes` | бейдж `чтение` / `чтение+запись` |
| Создан | `createdAt` | дата создания |
| Последнее использование | `lastUsedAt` | «5 минут назад» / «ни разу» — видно живые токены |
| Действие | — | кнопка «Отозвать» |

Пустое состояние: крупная кнопка «Создать токен» + пояснение.

### Блок 3 — Создание токена (главный флоу)
Кнопка «Создать токен» → модалка:
- **Название** (обязательно).
- **Права** (`scopes`): `Только чтение` / `Чтение и запись` (по умолчанию — запись).
- **Срок действия** (`expiresAt`, опционально): 30 / 90 дней / без срока.
- Кнопка «Создать».

После создания — экран с **полным токеном, показанным ОДИН раз**:
- Токен `mcp_…` крупно + кнопка «Копировать».
- Предупреждение: «Сохрани токен сейчас — больше он не отобразится» (в БД хранится только sha-256 хеш).

### Блок 4 — Готовый сниппет настройки MCP (киллер-фича)
Под токеном — готовый к вставке конфиг с уже подставленными значениями, вкладки под клиентов:

```json
{
  "mcpServers": {
    "botcraft-builder": {
      "command": "...",
      "env": {
        "API_BASE_URL": "https://<домен>",
        "MCP_AGENT_TOKEN": "mcp_…"
      }
    }
  }
}
```

Кнопка «Копировать конфиг». Превращает «прочитай доку и собери конфиг» в «скопировал → вставил → работает».

### Блок 5 — Отзыв
«Отозвать» → подтверждение → ставится `revokedAt` → токен мгновенно перестаёт работать (resolver проверяет `revokedAt IS NULL`). В списке помечается отозванным/исчезает.

## Под капотом (реализация 0.2b)

### Серверные хендлеры (под `requireAuth`, доступ только к своим токенам по `ownerId`)
- `POST /api/agent-tokens` — создать; возвращает **полный токен один раз** + запись. Тело: `{ label, scopes?, expiresAt? }`.
- `GET /api/agent-tokens` — список через **DTO** (без `tokenHash`/секрета): `id, label, prefix, scopes, createdAt, lastUsedAt, revokedAt`.
- `DELETE /api/agent-tokens/:id` — отозвать (только свой токен).

### Storage (уже готово в 0.2a)
- `createAgentToken(ownerId, label, scopes?)` → `{ token, record }`.
- `getAgentTokensByOwner(ownerId)`.
- `revokeAgentToken(id, ownerId)`.
- (`resolveAgentToken` — для рантайма, не для UI.)

### Включение scopes (рантайм-проверка)
- Read-токен → операции записи отклоняются. Где проверять: middleware/обёртка по HTTP-методу (GET = read; POST/PUT/PATCH/DELETE = write) либо на уровне тула. По умолчанию `scopes='read,write'`.

### Клиент
- Новая вкладка/страница + компоненты: список токенов, модалка создания, модалка «токен показан один раз» + копирование, сниппет конфига, отзыв. Запросы к трём эндпоинтам выше.

---

## Безопасность UX
- Полный токен показывается **строго один раз** при генерации; повторно — только префикс.
- В списке и DTO никогда не отдавать `tokenHash`/секрет.
- Отзыв необратим и мгновенен.
- Подтверждение перед отзывом.

---

## На будущее (не в первую версию)
- Индикатор «агент сейчас подключён/редактирует» (переиспользовать presence-бейдж холста).
- Лог последних действий агента (из истории версий по `authorKind='agent'`).
- Регенерация токена (отзыв + создание новым в один клик).
- Чек-сумма в префиксе токена (как `ghp_`+CRC у GitHub) для секрет-сканеров.
