# Broadcast Panel — Рассылки

> Массовая отправка сообщений пользователям бота с фильтрацией аудитории, live-прогрессом и историей.

---

## Идея

Сейчас диалог позволяет отправить сообщение только одному пользователю.
Broadcast — это отправка одного сообщения сразу всей базе или её сегменту.

Ключевые требования:
- Фильтрация аудитории (теги, дата регистрации, активность)
- Подстановка переменных `{first_name}`, `{username}` и т.д.
- Throttle 25 msg/sec — не превышать лимиты Telegram API
- Live-прогресс через WebSocket
- История рассылок с детальной статистикой
- Возможность остановить рассылку на ходу

## База данных

### Таблица `broadcasts`

```sql
CREATE TABLE broadcasts (
  id            serial PRIMARY KEY,
  project_id    integer NOT NULL REFERENCES bot_projects(id) ON DELETE CASCADE,
  token_id      integer NOT NULL DEFAULT 0,

  -- Содержимое
  name          text NOT NULL,                    -- название рассылки
  message_text  text NOT NULL,                    -- HTML-текст сообщения

  -- Фильтры аудитории (сохраняем для повтора)
  filters       jsonb NOT NULL DEFAULT '{}',
  -- {
  --   tags?: string[],
  --   registeredFrom?: string,   -- ISO date
  --   registeredTo?: string,
  --   activeFrom?: string,
  --   activeTo?: string
  -- }

  -- Статус
  status        text NOT NULL DEFAULT 'pending',
  -- 'pending' | 'running' | 'stopped' | 'done' | 'failed'

  -- Статистика
  total_count   integer NOT NULL DEFAULT 0,       -- всего получателей
  sent_count    integer NOT NULL DEFAULT 0,        -- отправлено
  delivered_count integer NOT NULL DEFAULT 0,     -- доставлено (ok от Telegram)
  failed_count  integer NOT NULL DEFAULT 0,        -- ошибок

  -- Время
  created_at    timestamp DEFAULT now(),
  started_at    timestamp,
  finished_at   timestamp
);
```

### Таблица `broadcast_results`

```sql
CREATE TABLE broadcast_results (
  id            serial PRIMARY KEY,
  broadcast_id  integer NOT NULL REFERENCES broadcasts(id) ON DELETE CASCADE,
  user_id       text NOT NULL,                    -- Telegram user_id
  status        text NOT NULL,
  -- 'sent' | 'failed' | 'blocked' | 'not_found'
  error_message text,                             -- описание ошибки от Telegram
  sent_at       timestamp DEFAULT now()
);

CREATE INDEX idx_broadcast_results_broadcast_id ON broadcast_results(broadcast_id);
CREATE INDEX idx_broadcast_results_status ON broadcast_results(broadcast_id, status);
```

## API

### Список рассылок
```
GET /api/projects/:projectId/broadcasts?tokenId=&page=&limit=
```
Ответ: `{ broadcasts: Broadcast[], total: number }`

### Создать и запустить рассылку
```
POST /api/projects/:projectId/broadcasts
Body: { name, messageText, filters, tokenId }
```
Запускает очередь отправки асинхронно. Возвращает `{ broadcastId }`.

### Детали рассылки
```
GET /api/projects/:projectId/broadcasts/:broadcastId
```
Ответ: `Broadcast` + `results: BroadcastResult[]` (только ошибки)

### Остановить рассылку
```
POST /api/projects/:projectId/broadcasts/:broadcastId/stop
```

### Предпросмотр аудитории
```
POST /api/projects/:projectId/broadcasts/preview-audience
Body: { filters, tokenId }
```
Ответ: `{ count: number, sample: UserBotData[] }` — количество + 3 примера пользователей

### Экспорт ошибок
```
GET /api/projects/:projectId/broadcasts/:broadcastId/errors.csv
```

## Серверная логика

### Очередь отправки

Telegram ограничивает: не более 30 сообщений/сек на бота.
Безопасный лимит — 25 msg/sec с паузой между батчами.

```
broadcastQueue(broadcastId):
  1. Загрузить всех пользователей по фильтрам (батчами по 100)
  2. Для каждого пользователя:
     a. replaceVariablesInText(messageText, userData)
     b. POST api.telegram.org/sendMessage
     c. Записать результат в broadcast_results
     d. Инкрементировать счётчики в broadcasts
     e. Отправить WS-событие broadcast-progress
     f. Пауза если превышен лимит (token bucket)
  3. Обновить status = 'done' / 'stopped'
  4. Отправить WS-событие broadcast-done
```

### WS-события

Новый тип события в `broadcastProjectEvent`:

```ts
// Прогресс отправки (каждые N сообщений или каждую секунду)
{
  type: 'broadcast-progress',
  projectId: number,
  broadcastId: number,
  sentCount: number,
  deliveredCount: number,
  failedCount: number,
  totalCount: number,
  status: 'running' | 'stopped' | 'done'
}
```

### Обработка ошибок Telegram

| Код ошибки | Причина | Действие |
|------------|---------|----------|
| 403 Forbidden | Пользователь заблокировал бота | Записать `blocked`, продолжить |
| 400 Bad Request: chat not found | Пользователь удалил аккаунт | Записать `not_found`, продолжить |
| 429 Too Many Requests | Превышен лимит | Подождать `retry_after` секунд |
| 5xx | Ошибка Telegram | Retry 3 раза с backoff |

## Технические риски

| Риск | Описание | Митигация |
|------|----------|-----------|
| Telegram rate limit | 30 msg/sec на бота | Throttle 25 msg/sec + обработка 429 |
| Долгая рассылка | 10k пользователей = ~7 минут | Фоновая очередь, WS-прогресс, кнопка стоп |
| Дубли при рестарте | Сервер упал в середине рассылки | Хранить `sent_user_ids` или статус по каждому пользователю в `broadcast_results` |
| Большая выборка | Загрузка 50k пользователей в память | Стриминг батчами по 100, не грузить всё сразу |
| Переменные с null | `{first_name}` = null у части пользователей | Fallback: пустая строка или `{first_name\|Пользователь}` |
