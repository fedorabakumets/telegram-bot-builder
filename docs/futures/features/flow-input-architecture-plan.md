# План: Новая Flow-модель для ввода и сохранения данных

## Контекст

Текущая модель смешивает в одной ноде несколько разных ролей:

- показ сообщения;
- показ клавиатуры;
- ожидание пользовательского ответа;
- валидацию ответа;
- сохранение ответа;
- переход к следующему шагу.

Это усложняет:

- UI нод;
- читаемость сценариев на canvas;
- генерацию кода;
- поддержку новых способов хранения данных;
- расширение под внешние интеграции.

Цель: перейти к более чистой flow-модели в духе `n8n`, где каждая нода делает одну понятную операцию.

## Базовая идея

Разделить логику на отдельные ноды:

1. `Trigger`
2. `Message`
3. `Keyboard`
4. `Wait For Input`
5. `Write Data`
6. `Condition`

При этом:

- `Condition` уже существует как отдельная нода и должна оставаться отдельной.
- `Message` должна только показывать текст.
- `Keyboard` должна только отображать кнопки.
- `Wait For Input` должна только ждать и валидировать пользовательский ответ.
- `Write Data` должна только сохранять данные в выбранное хранилище.

## Рекомендуемая цепочка

```text
Trigger
  ↓
Message
  ↓
Keyboard
  ↓
Wait For Input
  ↓
Write Data
  ↓
Condition
  ├─ true
  └─ false
```

Упрощённый пример:

```text
Text Trigger("приветик")
  ↓
Message("Сколько тебе лет?")
  ↓
Keyboard(14, 15, 16, 17)
  ↓
Wait For Input(type=text+button, validation=number)
  ↓
Write Data(input.value -> age)
  ↓
Condition(age < 16)
  ├─ Message("Не подходим")
  └─ Message("Продолжаем")
```

## Ноды

### 1. Trigger

Примеры:

- `Text Trigger`
- `Command Trigger`
- `Callback Trigger`

Назначение:

- запуск сценария.

### 2. Message

Назначение:

- показать текст;
- применить parse mode.

Что не должна делать:

- ждать ввод;
- сохранять ответ;
- валидировать ответ;
- управлять storage.

### 3. Keyboard

Назначение:

- показать inline/reply-клавиатуру;
- отрисовать кнопки и layout.

Что не должна делать:

- сохранять ответ;
- сама владеть `waiting_for_input`;
- принимать решение, куда писать данные.

### 4. Wait For Input

Назначение:

- ожидать ответ пользователя;
- валидировать ответ;
- возвращать нормализованный `input.value`.

Поддерживаемые типы:

- `text`
- `button`
- `photo`
- `video`
- `audio`
- `document`
- в перспективе: `contact`, `location`

Поля:

- `inputType`
- `sourceMode`
- `validationType`
- `min`
- `max`
- `regex`
- `retryMessage`
- `allowSkip`
- `timeout`

Выходы:

- `success`
- `invalid`
- `skip`
- `timeout`

### 5. Write Data

Назначение:

- записать результат ввода в выбранное хранилище.

Это лучше, чем отдельная узкая нода только для переменной, потому что позволяет масштабироваться под разные storage/provider.

Поля:

- `source`
- `targetType`
- `targetKey`
- `mode`
- `saveToDb`

Примеры `source`:

- `input.value`
- `input.photo`
- `input.document`
- `expression`

Примеры `mode`:

- `replace`
- `append`

## Хранилища для Write Data

### 1. Variable

Пример:

```text
input.value -> age
```

Использование:

- локальная переменная сценария;
- пользовательские переменные проекта.

### 2. Project Database

Пример:

```text
input.value -> users.age
```

Использование:

- стандартная запись в БД проекта;
- сохранение данных пользователя для условий и повторного использования.

### 3. Google Sheets

Пример:

```text
input.value -> Google Sheets / users / column age
```

Использование:

- запись лидов;
- экспорт анкет;
- простая интеграция с таблицами.

### 4. Airtable

Airtable — внешний сервис, похожий на “умную таблицу / лёгкую базу данных”.

Пример:

```text
input.value -> Airtable / table Users / field age
```

Использование:

- CRM-подобные сценарии;
- каталоги;
- no-code backend.

### 5. Webhook / API

Webhook / API — отправка данных во внешний сервис по HTTP.

Пример:

```text
POST https://example.com/leads
{
  "age": input.value
}
```

Использование:

- интеграция с CRM;
- отправка данных на свой сервер;
- запуск внешней автоматизации.

## Почему `Write Data`, а не только `Set Variable`

Если сделать только `Set Variable`, архитектура быстро упрётся в необходимость:

- отдельно писать в БД;
- отдельно писать в Google Sheets;
- отдельно отправлять webhook.

Одна универсальная нода `Write Data` лучше, потому что:

- единая mental model;
- один UI-паттерн;
- легко расширять storage providers;
- меньше размножения похожих нод.

## Рекомендуемый MVP

Минимальный этап внедрения:

1. Добавить `Wait For Input`
2. Добавить `Write Data`
3. Сохранить `Message`, `Keyboard`, `Condition` как отдельные ноды
4. Поддержать только два storage provider:
   - `Variable`
   - `Project Database`

## Второй этап

После стабилизации MVP:

- `Google Sheets`
- `Webhook / API`

## Третий этап

После этого:

- `Airtable`
- другие внешние интеграции

## Что нужно убрать из старой модели

Из `message/start/command` желательно вынести:

- `collectUserInput`
- `enableTextInput`
- `enablePhotoInput`
- `enableVideoInput`
- `enableAudioInput`
- `enableDocumentInput`
- `inputVariable`
- `photoInputVariable`
- `videoInputVariable`
- `audioInputVariable`
- `documentInputVariable`
- логику `waiting_for_input`
- логику выбора storage

## Итоговая архитектурная мысль

Flow должен читаться как последовательность явных действий:

```text
Показать вопрос
→ Дождаться ответа
→ Записать данные
→ Проверить условие
→ Пойти по нужной ветке
```

Это лучше текущей смешанной модели и ближе к понятной automation-архитектуре в духе `n8n`.
