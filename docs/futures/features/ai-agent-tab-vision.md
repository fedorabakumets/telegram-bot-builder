# Вкладка ИИ-агента и MCP-сервер конструктора

Документ описывает целевую архитектуру встроенного ИИ-агента, который собирает и редактирует ботов прямо в конструкторе, а также MCP-сервера, открывающего те же возможности внешним ИИ-клиентам (Kiro, Claude Desktop). Это видение («северная звезда»), а не план ближайшего спринта — реальный старт берёт малое подмножество. Приоритет слоёв: 🔴 фундамент → 🟡 расширение → 🟢 на потом.

> **Реализовано (MCP):** внешний MCP-сервер `botcraft-builder` — слои 1–3 и часть слоя 4. Документация: [[mcp/bot-builder]], обзор: [[mcp/overview]]. Код: `lib/bot-tools/`, `tools/mcp-server/`.

## Главная идея

Сейчас при генерации `project.json` внешний ИИ галлюцинирует структуру нод (классика — `condition` с несуществующим `conditions` + `defaultTarget` вместо `branches`). Цель — сделать это **невозможным**: на каждом шаге правду диктует код конструктора (zod-схемы, генератор, валидатор), а не догадки модели.

Ключевое наблюдение: **вкладка ИИ-агента и MCP-сервер — это два «лица» одного слоя инструментов.** Тулы (валидация, генерация, `create_node`, `auto_layout`) пишутся один раз и переиспользуются обоими. Делать их дважды не нужно.

```
        ┌─────────────────────────────────────┐
        │   Слой инструментов (ядро)           │
        │   lib/ + server/services/bot-tools/  │
        │   validate / generate / create_node  │
        │   connect_nodes / auto_layout / ...  │
        └───────────────┬─────────────────────┘
                        │ один источник правды
            ┌───────────┴────────────┐
            ▼                         ▼
   ┌──────────────────┐     ┌──────────────────────┐
   │ MCP-сервер        │     │ Вкладка ИИ-агента    │
   │ (внешние ИИ:      │     │ (внутри конструктора,│
   │  Kiro/Claude)     │     │  свой LLM-бэкенд)    │
   └──────────────────┘     └──────────────────────┘
```

## Слой 1 — Знание формата (introspection)

Чтобы ИИ не выдумывал, а спрашивал. Read-only, нулевой риск.

- 🔴 `list_node_types` — все типы нод с кратким описанием и назначением.
- 🔴 `get_node_schema(type)` — полная схема `data` ноды: поля, типы, дефолты, обязательность. Источник — `shared/schema/tables/node-schema.ts`.
- 🔴 `list_operators` — допустимые операторы condition-ноды (белый список из `AGENTS.md`: `filled`, `equals`, `is_admin` и т.д.), чтобы исключить `not_empty` и прочую отсебятину.
- 🟡 `get_node_example(type)` — рабочий пример `data` из эталонной ноды (правило «копировать формат из рабочих нод»).
- 🟡 `list_commands` — стандартные команды (`StandardCommand` из `lib/commands`).
- 🟡 `get_prompt_guide` — отдать актуальный `docs/bot-json-prompt.md`, чтобы клиент подгрузил правила формата.

## Слой 2 — Валидация (главная ценность)

- 🔴 `validate_bot_project(project_json)` — структурная проверка через zod (`shared/schema`) + доменные правила: операторы, целостность `target`-ссылок, обязательные поля. Возврат — ошибки с путём до поля и человекочитаемым объяснением.
- 🟡 `validate_node(type, data)` — проверка одной ноды при пошаговой сборке.
- 🟡 `lint_bot_project(project_json)` — мягкие предупреждения: висячие ноды, недостижимые ветки, переменная в condition без заполнения, дубли id, пустой `start`.

Требование: валидатор — единственный источник правды, переиспользует ровно тот же код, что и UI, чтобы вердикты MCP и интерфейса не расходились.

## Слой 3 — Генерация

- 🔴 `generate_bot_code(project_json)` — готовый `bot.py` через `generatePythonCode`, прогнанный через `assertValidPython`.
- 🟡 `generate_project_files(project_json)` — полный каркас: `bot.py` + `requirements.txt` + `Dockerfile` + `README` + `.env.example` (есть `generateRequirementsTxt`, `generateDockerfile`, `generateReadme`, `generateEnvFile`).
- 🟢 `explain_generated_code(project_json, node_id)` — какой кусок Python соответствует ноде (через source-map `CodeWithMap` / `CodeNodeRange`).

## Слой 4 — Конструирование

Сервер из «валидатора» превращается в «руки» для ИИ.

- 🟡 `create_node(type, partial_data)` — собрать корректную ноду: дополнить дефолтами, проставить структуру (для condition — сразу правильные `branches` с `else`).
- 🟡 `connect_nodes(project, from_id, to_id, branch?)` — добавить переход без ручного ковыряния `target`.
- 🟡 `add_node` / `update_node` / `remove_node` — операции над `project.json` с автопроверкой целостности.
- 🟡 `auto_layout(project)` — расставить координаты нод. Логику из `client/utils/hierarchical-layout.ts` стоит вынести в общий модуль для переиспользования.
- 🟢 `scaffold_from_description(text)` — из текста собрать черновой валидный `project.json` (связка слоёв 1–4).

## Слой 5 — Жизненный цикл проекта (stateful, с подтверждением)

- 🟡 `list_bots` / `get_bot(id)` — читать существующие проекты (из `bots/` и/или БД).
- 🟢 `save_bot` / `create_bot` / `clone_bot` — писать. Опасно, требует подтверждения.
- 🟢 `diff_bots(a, b)` — сравнить проекты (есть `tools/compare_bots/`).
- 🟢 `export_bot` / `import_bot` — обмен `project.json`.

## Слой 6 — Рантайм и деплой (максимальный риск)

- 🟢 `dry_run_bot(project)` — прогнать сгенерированного бота в песочнице без реального токена.
- 🟢 `start_bot` / `stop_bot` / `get_bot_logs` — управление запуском (`server/bots/`, `server/terminal/`). Только с подтверждением.
- 🟢 `deploy_bot` — деплой (Railway-конфиги уже есть). Самый опасный, в самый конец.

## Вкладка ИИ-агента в UI

Встаёт рядом с переключателем «Холст / JSON» (третий режим рабочей области либо боковая чат-панель поверх холста). Внутри — диалог на русском, где агент:

1. Собирает бота с нуля по описанию (тулы → `auto_layout` → рисует на холсте).
2. Редактирует текущий проект точечно («добавь кнопку Назад на каждом экране»).
3. Объясняет и отлаживает («почему бот не доходит до оплаты?», «покажи Python этого блока»).
4. Сам чинит ошибки через `validate` / `lint`.
5. Показывает превью и diff перед применением.

Критично заложить сразу:

- 🔴 **Применение через те же команды, что и ручное редактирование** (add/update/remove/connect node), чтобы работали undo/redo и история. Агент не должен писать `project.json` в обход состояния.
- 🔴 **Превью + подтверждение перед применением** — особенно для массовых изменений (модель «Supervised»).
- 🟡 **Стриминг ответа** — потоковая отдача токенов (SSE/WebSocket; `ws` уже в зависимостях).
- 🔴 **Ключ к LLM на бэкенде, не на клиенте** — запросы идут через `server/`, ключ в env. Клиент общается только со своим эндпойнтом (безопасность + учёт лимитов).

### Прикидка по задачам

Агент работает в цикле «модель → вызов тула → снова модель», поэтому за задачу проходит несколько запросов.

**Сборка целого бота с нуля** (~100K токенов входа суммарно + ~15K выхода):

| Модель | Стоимость задачи |
|---|---|
| DeepSeek | ~$0.02 |
| Gemini Flash | ~$0.07 |
| Claude Haiku | ~$0.18 |
| Claude Sonnet | ~$0.53 |
| GPT-5.5 / Opus | ~$0.9–1.0 |

**Мелкая правка** («добавь кнопку», «поменяй условие») — в 4–5 раз дешевле.

**На активного пользователя в месяц** (~100 действий):

| Класс моделей | Бюджет/мес |
|---|---|
| Бюджетные | ~$1–5 |
| Средние (Haiku/Sonnet) | ~$10–40 |
| Топ (GPT-5.5/Opus) | ~$50–90 |

### Что режет счёт

- **Prompt caching** — системный промпт + `docs/bot-json-prompt.md` + схемы нод одинаковы на каждом запросе. Кэш даёт −50…90% на входных токенах. Для агента критично: вход тяжёлый (схемы, контекст графа), выход лёгкий.
- **Роутинг моделей** — мелкие правки на дешёвой модели (DeepSeek/Haiku), сложную сборку на средней.
- **Тулы вместо «думания»** — валидацию/генерацию делает код, а не модель; меньше токенов на размышления, точнее результат.

### Вывод по выбору модели

Топовая модель не нужна: задача структурная, тяжёлую работу делают тулы.
- **Старт/MVP:** Gemini Flash или DeepSeek — копейки, удобно отлаживать агента.
- **Прод:** Claude Sonnet 4.6 как основная + Haiku/DeepSeek на мелочь через роутинг.

Реалистичный операционный бюджет на активного пользователя — **~$5–20/мес** при разумном кэшировании и роутинге.

## Порядок реализации

1. **Слой инструментов** (валидация + генерация + introspection поверх `lib`/`shared`) — фундамент для обоих лиц.
2. **MCP-сервер** — тонкая stdio-обёртка над слоем. Быстро, изолированно, тестируется без UI; служит полигоном для отладки тулов.
3. **Бэкенд агента** — эндпойнт в `server/` (LLM + tool-calling по тому же слою + стриминг).
4. **Вкладка-чат в UI** — `client/components/editor/`, применение изменений через существующие команды графа.

Рекомендуемый старт: слои 1–3 (знание + валидация + генерация). Это ~80% пользы, чистый read-only, нулевой риск, и сразу убирает главную боль с кривым `project.json`. Слои 4–6 наращиваются итеративно.

## Открытые вопросы

- **LLM-провайдер для агента** — один провайдер или абстракция под несколько (для роутинга и резервирования).
- **Транспорт MCP-сервера** — stdio (локальный процесс для Kiro/Claude Desktop) против HTTP/SSE (удалённый доступ). По умолчанию — stdio.
- **Формат вкладки** — отдельный режим рабочей области (как «Холст / JSON») или плавающая боковая чат-панель.
- **Вынос `auto_layout`** — где разместить общий модуль раскладки, чтобы переиспользовали и UI, и тулы.
