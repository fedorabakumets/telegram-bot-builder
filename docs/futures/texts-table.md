# Фича: Системная таблица текстов (_texts)

## Концепция

Автоматическая таблица `_texts` на уровне проекта, которая содержит все тексты сообщений из нод сценария. Двусторонняя синхронизация: изменил в ноде → обновилось в таблице, изменил в таблице → обновилось в ноде. Бот при работе читает тексты из таблицы (с кэшем), что позволяет менять тексты без перезапуска.

## Проблема

- Тексты зашиты в JSON сценария — чтобы изменить одно слово, нужно открыть ноду, найти текст, отредактировать, сохранить, пересобрать бота
- Нет единого места где видны все тексты бота
- Не-программисты не могут менять тексты без понимания визуального редактора
- Нет горячей перезагрузки текстов без рестарта бота

## Решение

### Таблица `_texts`

Специальная пользовательская таблица (bot_table) с зарезервированным именем `_texts`. Создаётся автоматически при создании проекта.

Структура:
| Колонка | Описание |
|---------|----------|
| key | ID ноды (уникальный ключ) |
| sheet | Имя листа (для группировки) |
| label | Человекочитаемое описание (из description ноды или первые 30 символов) |
| value | Текст сообщения (с переменными {first_name} и т.д.) |

### Синхронизация: Сценарий → Таблица

При сохранении проекта (PUT /api/projects/:id) сервер:
1. Проходит по всем нодам типа `message`, `photo`, `video`, `audio`, `document` (всё что имеет `messageText` или `mediaCaption`)
2. Для каждой ноды с непустым текстом — upsert в `_texts`:
   - key = node_id
   - sheet = имя листа
   - label = первые 30 символов текста или description ноды
   - value = полный текст (messageText / mediaCaption)
3. Удаляет из `_texts` строки, чьи key не найдены в сценарии (нода удалена)

### Синхронизация: Таблица → Сценарий

При изменении строки в `_texts` (через API таблиц):
1. Сервер находит ноду по `key` (node_id) в JSON сценария
2. Обновляет `messageText` / `mediaCaption` в ноде
3. Сохраняет проект

### Бот: чтение текстов из БД

В сгенерированном Python-коде:

```python
import asyncio
from typing import Dict

# Кэш текстов
_texts_cache: Dict[str, str] = {}
_texts_table_id: int | None = None

async def load_texts(pool):
    """Загрузка текстов из таблицы _texts при старте"""
    global _texts_table_id
    row = await pool.fetchrow(
        "SELECT id FROM bot_tables WHERE project_id = $1 AND name = '_texts'",
        PROJECT_ID
    )
    if not row:
        return
    _texts_table_id = row["id"]
    await reload_texts(pool)

async def reload_texts(pool):
    """Перезагрузка кэша текстов"""
    if not _texts_table_id:
        return
    rows = await pool.fetch(
        "SELECT data FROM bot_table_rows WHERE table_id = $1",
        _texts_table_id
    )
    _texts_cache.clear()
    for r in rows:
        data = r["data"]
        if data.get("key") and data.get("value"):
            _texts_cache[data["key"]] = data["value"]

def get_text(node_id: str, fallback: str = "") -> str:
    """Получить текст по ID ноды (с fallback на захардкоженный)"""
    return _texts_cache.get(node_id, fallback)

# Периодическая перезагрузка (каждые 60 сек)
async def texts_reload_loop(pool):
    while True:
        await asyncio.sleep(60)
        await reload_texts(pool)
```

В шаблоне отправки сообщения:
```python
# Вместо:
# text = "👋 Привет, {first_name}!"

# Теперь:
text = get_text("msg-welcome", "👋 Привет, {first_name}!")
text = resolve_variables(text, variables)
```

Fallback гарантирует что бот работает даже если таблица пуста или удалена.

## UI

### Вкладка "Таблицы"

Таблица `_texts` отображается в секции "Пользовательские (проект)" с особой иконкой:
```
ПОЛЬЗОВАТЕЛЬСКИЕ (ПРОЕКТ)
  ✏️ Тексты (авто)     12
  📝 profiles
  📝 relationships
```

Отличия от обычных таблиц:
- Нельзя удалить таблицу
- Нельзя добавлять/удалять колонки
- Можно добавлять строки вручную (для текстов не привязанных к нодам)
- Колонка `value` — расширенный редактор (многострочный, с подсветкой переменных)
- Колонка `key` — read-only (генерируется автоматически)

### Редактор ноды

В панели свойств ноды `message` рядом с полем текста — индикатор:
- 🔗 "Синхронизировано с таблицей текстов" (если текст есть в _texts)
- При редактировании текста в ноде — таблица обновляется при сохранении проекта

## Порядок реализации

### Фаза 1: Бэкенд (синхронизация сценарий → таблица)

1. При создании проекта — автоматически создавать таблицу `_texts` с колонками (key, sheet, label, value)
2. При сохранении проекта (PUT /api/projects/:id) — синхронизировать тексты из нод в `_texts`
3. API: при обновлении строки в `_texts` — обновлять messageText в JSON сценария

### Фаза 2: Генератор бота

4. В шаблоне бота — добавить загрузку текстов из `_texts` при старте
5. Использовать `get_text(node_id, fallback)` вместо хардкода
6. Фоновая перезагрузка каждые 60 секунд

### Фаза 3: UI

7. Отображение `_texts` как особой таблицы в списке
8. Расширенный редактор для колонки `value`
9. Индикатор синхронизации в панели свойств ноды

## Ограничения

- Только ноды с `messageText` и `mediaCaption` синхронизируются
- Тексты кнопок (`buttons[].text`) пока не синхронизируются (будущее)
- Максимум 500 текстов на проект (ограничение для производительности кэша)
- Ключ = node_id, поэтому при дублировании ноды создаётся новая запись

## Совместимость

- Старые проекты без `_texts` работают как раньше (fallback на хардкод)
- Таблица создаётся при первом сохранении проекта после обновления
- Генератор бота проверяет наличие таблицы — если нет, работает по-старому
