# Roadmap: MCP как наблюдательно-управляющий слой над платформой

Документ описывает план, при котором MCP-сервер (ассистент) получает доступ
**ко всему, что связано с конструктором**: логам ботов, аналитике, пользовательским
таблицам, статусам и переменным — чтобы не только строить сценарий, но и видеть,
как бот работает, и помогать его отлаживать и эксплуатировать.

> Редактирование сценария в реальном времени описано отдельно в
> [`mcp-live-editing.md`](./mcp-live-editing.md). Здесь — про **чтение/наблюдение**
> и (с ограничениями) **управление** всей платформой.

## Идея

Сейчас MCP умеет работать только со структурой сценария (ноды в `project.json` / БД).
Цель — превратить его в **копилота над всей платформой**: на вопрос «почему бот не
отвечает на /start» или «где в воронке теряются пользователи» агент сам посмотрит
логи и аналитику, а не будет гадать.

Принцип: **сначала read-слой (наблюдаемость) — он почти бесплатный и сразу полезен,
потом control-слой (управление) — аккуратно, с правами и защитой секретов/PII**.

## Две роли (важно разделять)

### 1. Наблюдение (read-only) — низкий риск, делать первым
| Область | Что даёт | Источник данных |
|---------|----------|-----------------|
| Логи ботов | Что бот делал, где упал, ошибки | `bot_logs`, история запусков `bot_launch_history` |
| Аналитика | Пользователи, сообщения, воронки, отвалы | сервисы статистики, `referral-funnel-analysis` |
| Таблицы | Что реально лежит в данных бота | `bot_tables` / `bot_table_columns` / `bot_table_rows` |
| Пользователи | Аудитория бота | `bot_users`, `telegram_users` |
| Состояние | Статусы ботов, инстансы, env (без значений секретов) | `bot_instances`, `worker_processes`, `bot_env_variables` |

### 2. Управление (write/control) — выше риск, с ограничениями
| Действие | Риск | Защита |
|----------|------|--------|
| Редактирование сценария | средний | версии/чекпоинты (см. mcp-live-editing) |
| Запуск/стоп/рестарт бота | средний | подтверждение, scope по проекту |
| Правка таблиц / переменных | средний–высокий | scope, аудит, защита секретов |
| Откат к версии/чекпоинту | низкий | уже есть в версионировании |

## Обязательные ограничения (иначе будут проблемы)

- **Секреты не читаем.** Токены ботов, ключи, значения `.env` — агент НЕ получает
  их значения. Для env отдаём **только имена ключей**, не значения.
- **PII под контролем.** Таблицы и `telegram_users` — персональные данные реальных
  людей. Минимум: scoping по проекту; в проде — явное разрешение и аудит доступа.
- **Права и scope.** «MCP видит всё» допустимо локально (`SKIP_AUTH`), но в проде
  агент работает строго в рамках конкретного проекта/владельца, а не всей БД.
- **Объём.** Логи и строки таблиц бывают огромными — обязательны пагинация, фильтры
  и агрегаты, а не «выгрузи всё целиком».

## Предлагаемые инструменты MCP (read-слой)

Тонкие обёртки поверх уже существующих сервисов/`storage` — не новый бэкенд:

| Инструмент | Назначение | Параметры (примерно) |
|------------|------------|----------------------|
| `get_bot_logs` | Логи бота с фильтром по уровню/времени | projectId, level?, since?, limit, offset |
| `get_launch_history` | История запусков/остановок | projectId, limit |
| `get_analytics` | Сводная аналитика (юзеры/сообщения/воронка) | projectId, range |
| `query_table` | Чтение строк пользовательской таблицы | projectId, tableId, filter?, limit, offset |
| `list_bot_users` | Аудитория бота | projectId, filter?, limit, offset |
| `get_bot_status` | Статусы ботов и инстансов | projectId |
| `get_env_keys` | **Имена** переменных окружения (без значений) | projectId |

## План реализации

### Фаза 1 — Read-слой (наблюдаемость)
| # | Задача | Сложность | Статус |
|---|--------|-----------|--------|
| 1.1 | `get_bot_logs` + `get_launch_history` (пагинация, фильтры) | Низкая–средняя | ⬜ |
| 1.2 | `get_analytics` (сводка по воронке/сообщениям) | Средняя | ⬜ |
| 1.3 | `query_table` + `list_bot_users` (пагинация, фильтры, без PII-утечки наружу) | Средняя | ⬜ |
| 1.4 | `get_bot_status` + `get_env_keys` (только имена ключей) | Низкая | ⬜ |
| 1.5 | Scoping по проекту/владельцу + базовый аудит доступа | Средняя | ⬜ |

### Фаза 2 — Control-слой (управление)
| # | Задача | Сложность | Статус |
|---|--------|-----------|--------|
| 2.1 | Запуск/стоп/рестарт бота через MCP (с подтверждением) | Средняя | ⬜ |
| 2.2 | Правка таблиц/переменных (с аудитом, защита секретов) | Средняя–высокая | ⬜ |
| 2.3 | Откат к версии/чекпоинту через MCP | Низкая | ⬜ |

### Фаза 3 — Связка с диагностикой
| # | Задача | Сложность | Статус |
|---|--------|-----------|--------|
| 3.1 | Агент сам коррелирует логи + аналитику + сценарий для ответа «почему сломалось» | Средняя | ⬜ |
| 3.2 | Подсветка проблемной ноды на холсте по данным из логов | Средняя | ⬜ |

## Рекомендуемая последовательность

1. **Фаза 1 (read)** — почти бесплатно (обёртки над существующими данными) и сразу
   делает агента полезным для отладки. Начать с логов и статусов.
2. **Фаза 2 (control)** — добавляем поштучно, каждое действие с правами/подтверждением.
3. **Фаза 3** — «умная» диагностика поверх собранных данных.

## Открытые вопросы

- Где запускается MCP относительно сервера и БД (локально / в сети) — влияет на права.
- Единый ли scope с live-редактированием (см. `mcp-live-editing.md`) или отдельный.
- Как маскировать PII в выдаче агента, если результаты уходят во внешнюю модель.
