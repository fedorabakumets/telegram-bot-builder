# Telegram Client API: Триггеры И Действия

Этот документ описывает отдельный продуктовый слой для работы через Telegram Client API, а не через обычный Bot API.

Client API подходит для сценариев, где система должна не только отвечать пользователю как бот, но и читать историю чатов, наблюдать за событиями, анализировать участников, собирать сообщения, выполнять синхронизацию и строить парсер поверх реальных чатов, групп и каналов.

## Чем Client API Отличается От Bot API

Bot API лучше подходит для:
- ответа пользователям от имени бота
- кнопок, меню и сценариев диалога
- простых webhook-driven интеграций

Client API лучше подходит для:
- чтения истории чатов
- мониторинга групп и каналов
- извлечения участников
- анализа сообщений и медиа
- парсинга переписок
- синхронизации состояния чата

Практически это означает, что в продукте имеет смысл отделять:
- `Bot API` сценарии
- `Client API` сценарии
- гибридные сценарии, где чтение идёт через Client API, а действия ответа или уведомления через Bot API

- `Сравнить состав группы`

- `Транскрибировать голосовое`
- `Записать в базу`
- `Отправить в Webhook`

### Требует Осторожной Формулировки

- `Получить всех пользователей группы`
- `Получить всех ушедших участников`
- `Получить всю историю без ограничений`
- `Мгновенно синхронизировать большой канал`
- `Гарантированно извлечь все медиа и весь контент`

Такие формулировки лучше заменять на более точные:
- `Получить доступных участников`
- `Определить выбывших участников по сравнению со снимком`
- `Синхронизировать доступную историю`
- `Загрузить последние сообщения`

## Рекомендация По UI

В интерфейсе лучше не смешивать узлы Client API и Bot API в одну плоскую кучу.

Практичный вариант:
- отдельная категория `Telegram Bot API`
- отдельная категория `Telegram Client API`
- отдельная категория `Парсинг и аналитика`

Так пользователю будет понятнее, что:
- один набор узлов работает как бот
- другой набор читает и анализирует чаты
- а parser-слой может использовать данные из обоих источников
